Amiga Vertical Blanking (VBlank) — Explanation + Interactive Demo

What VBlank is, why you use it on the Amiga, how to set up a vertical-blank interrupt, and a simple interactive simulation.

What is Vertical Blanking?

Vertical blanking (VBlank) is the short period when a CRT electron beam moves from the bottom of the screen back to the top to begin a new frame. During VBlank the display hardware is not actively scanning visible lines, so it's the ideal moment to update the frame buffer without producing tearing or visible artifacts.

Why use the VBlank interrupt on the Amiga?

  • Swap framebuffers (double buffering) without tearing.
  • Update playfield pointers, palette changes, or Copper lists safely.
  • Perform per-frame game logic synchronized to refresh rate.

Common approaches to synchronize to VBlank

From simplest to most sophisticated:

  1. Busy-wait — poll a scanline/vertical position register until VBlank begins. Simple but wastes CPU cycles.
  2. Copper wait — use the Copper co-processor to wait for a raster line and trigger actions (good for timed register changes).
  3. VBlank interrupt — install an interrupt handler that the custom chips fire each VBlank. Allows CPU to do other work and react when the frame boundary occurs.

How the VBlank interrupt works (conceptual)

When the raster reaches the vertical blank region, the Amiga's custom circuitry raises an interrupt. The OS/hardware has an interrupt controller that can deliver this event to the CPU. A program installs a handler (an interrupt service routine) and enables the VBlank source. On each VBlank the CPU jumps to the handler — inside the handler you must quickly acknowledge the interrupt, do minimal work (flip buffers / signal higher-level code), and return.

Handler responsibilities & good practices

  • Acknowledge the interrupt early (clear or write the required bits) so it doesn't retrigger immediately.
  • Do not perform heavy work inside the interrupt. Instead set a flag/queue and handle the heavy work in the main loop.
  • Save and restore registers and chain to the previous handler if required by your program environment.
  • Keep the handler short and deterministic to avoid missing future interrupts.

Example: pseudocode (assembly-like) for a minimal VBlank handler

; PSEUDOCODE — illustrative only
; InstallHandler:
;   - Save old handler address
;   - Write address of MyVBlankHandler into system interrupt vector
;   - Enable VBlank interrupt source
;
; MyVBlankHandler:
;   push all used registers
;   acknowledge VBlank (clear interrupt flag)
;   inc.w VBlankCounter      ; lightweight work
;   set a flag to tell main loop to swap buffers
;   pop all registers
;   return from interrupt
;

This is intentionally high-level: exact vector locations and acknowledgement specifics depend on the system/Kickstart and whether you're writing a boot ROM driver, a resident program, or running from the OS.

Alternative: waiting for VBlank in your main loop (busy-wait)

WaitVBlank:
  ; read a raster/position register until it indicates VBlank
  ; return when VBlank begins

Busy-waiting is easy but prevents the CPU from doing other useful work while waiting.
